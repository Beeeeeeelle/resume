Module 0

Architecture Split

`Ask Belle` 的第一课不是界面，而是边界。因为主站是静态 Jekyll 站，前端 widget 可以直接放站内， 但模型调用必须放到独立后端。这个架构决定了后面所有实现细节。

Jekyll Static Site Serverless API Grounding
1. Site Config

在 _config.yml 里定义 chatbot 的 title、button label、suggested prompts、 API 地址和 knowledge base 地址。

2. Widget Markup

用 _includes/chatbot-widget.html 把 launcher、panel、messages、suggestions、 composer 作为站内 UI 结构注入。

3. Behavior Layer

assets/js/chatbot.js 负责 open/close、focus、send、typing、fallback、scroll anchor 和 citations 渲染。

4. Serverless Backend

chatbot-backend/api/chat.js 做 CORS、限流、知识匹配、OpenAI 请求和 grounded instructions。

Module 1

Implementation Flow

这一部分把 Ask Belle 的真实 build sequence 拆成五步。重点不是“写了哪些文件”，而是每一步为什么必须这样做。

01

先定 assistant 的边界

不是做开放式 chatbot，而是做一个 bounded site assistant。它只回答 Belle 网站上已经公开的 research、 publications、teaching、projects 相关问题，并允许把联系类问题引导到公开链接。

02

把 UI 作为站内组件注入

用 include 模式把 launcher、panel、messages 和 composer 放进站点 footer 区域，避免重构现有导航和页面模板。

03

前端做出“产品级行为”

真正费时间的不是发送文本，而是焦点管理、点击外部关闭、Esc、loading state、fallback mode、滚动锚点和移动端布局。

04

后端单独部署并加安全边界

模型调用不能放前端。后端需要处理 ALLOWED_ORIGIN、rate limiting、body size、timeout、knowledge matching、 concise instructions 和环境变量。

05

上线后继续修真实 UI bug

缓存、遮挡、z-index、错误的 scroll target、被拉伸的 contact chips，这些问题都只有在真实页面和真实视口里才会暴露。

Module 1.5

Build Your Own Version

如果你不是来“看 Belle 做了什么”，而是想照着做一个自己的版本，这一节就是操作说明。默认假设你已经有一个静态站， 想加一个只回答站内公开内容的 AI assistant。

1

先写清楚它能回答什么

不要先接模型。先写 4 个允许回答的主题，1 句超出范围时的收口语。对个人网站来说，最稳的范围通常是 bio、projects、publications、teaching、contact pathways。

Output: scope statement Files: notes or config draft
  • 列出允许回答的栏目。
  • 列出禁止臆测的信息，如未公开经历、私人联系方式、未发布项目。
  • 写一句 fallback 文案，明确“我只能根据站内内容回答”。
2

先做一个小知识库，而不是直接把整站扔给模型

准备一个轻量 JSON 文件，里面只放你希望 assistant 优先检索的页面摘要。这样更容易控制回答边界，也方便 fallback mode 复用。

Output: curated JSON Files: /assets/data/site-knowledge.json
  • 每条记录至少有 title、url、summary。
  • 可选加 keywords，便于前端或后端做轻量匹配。
  • 只收录你愿意被引用的公开页面。
3

给站点加一层显式配置

不要把按钮文案、API 地址、suggested prompts 写死在脚本里。把这些放进站点配置，让内容边界和 UI 策略能单独调整。

Output: chatbot config block Files: _config.yml or site settings
  • 至少配置 title、subtitle、button label、api_url、knowledge_base_url。
  • 加 3 到 4 个 suggested prompts，直接暗示使用范围。
  • 把 feature toggle 也放进去，方便先本地试跑。
4

把 chat UI 当成站内组件，而不是单独起一个 app

用 include、partial 或组件方式插进现有网站。结构上至少要有 launcher、panel、messages、suggestions 和 composer。

Output: injected widget Files: include + CSS + JS entry
  • 先让它能打开和关闭，再考虑模型接入。
  • 先把 fallback 模式的静态回复跑通。
  • 确保 widget 不会和 masthead、search、footer 冲突。
5

前端优先做“行为”，不是先追求答案质量

一开始最容易忽略的是交互：Esc、click-outside、focus、loading、错误态、scroll anchor、mobile layout。 这些会直接决定它像不像一个产品组件。

Output: usable interaction loop Files: main chat script
  • 先支持 open / close / send / reset。
  • Live AI 出错时自动退回 knowledge-base fallback。
  • 回答生成后，滚到用户能看到自己问题的位置。
6

把模型调用放到独立后端

静态站里不能放密钥。最小后端要处理：来源限制、消息清洗、轻量检索、模型调用、错误收口、限流。

Output: /api/chat Files: serverless function + README
  • 只允许你的站点 origin。
  • 限制 payload 大小、消息条数、请求频率。
  • 把知识库结果和当前页面上下文一并传给模型。
7

最后才做部署连接和 smoke test

部署不是结束，而是验证阶段。你要同时确认前端缓存、后端环境变量、计费额度、CORS 和真实页面布局。

Output: live feature Files: deployment config + env vars
  • 先本地 build，再上线 backend，再回填 api_url。
  • 测一遍 live AI、fallback、移动端、缩小窗口和刷新缓存。
  • 准备一个 launch checklist，避免“接口通了但体验坏了”。

Done means

别人真正做完，不是“有个聊天框”，而是下面这几条都成立：

  • 能从主站打开和关闭。
  • Live AI 失败时不会整块坏掉。
  • 回答不超出站内公开内容。
  • 每次回答后，用户能看到自己刚问的问题和答案开头。
  • 窄视口下不遮导航，不挤爆布局。

Teaching Modules

这是一套更像 workshop / mini-course 的课程结构。每个模块都对应 repo 里的真实实现面，而不是抽象空谈。

Module A — Scope the Assistant

先决定 assistant 回答什么、不回答什么，以及为什么 fallback 比“全挂掉”更适合个人网站。

_config.yml assets/data/belle-site-knowledge.json
  • bounded assistant vs. open-ended chat
  • suggested prompts as scope cues
  • public links as safe contact surface

Module B — Build the Widget

把聊天入口作为现有网站的一部分，而不是新起一个 app shell。组件结构越清楚，后面交互修起来越快。

_includes/chatbot-widget.html assets/css/main.scss
  • launcher / panel / messages / composer decomposition
  • contacts and suggestions as UX accelerators
  • layout choices that avoid fighting the masthead

Module C — Make the Frontend Behave

这里是“接模型之外”的大头工程：状态管理、焦点控制、关闭逻辑、typing state、citations 和 scroll anchoring。

assets/js/chatbot.js
  • click-outside and Esc behavior
  • fallback mode when live AI fails
  • why scroll bugs are easy to get subtly wrong

Module D — Deploy a Safe Backend

静态站接 AI 的真正边界在后端。这里不是只会发请求，而是要做 prompt shaping、retrieval、timeouts 和 origin controls。

chatbot-backend/api/chat.js chatbot-backend/README.md

Module 3

Bug Fixes That Actually Teach Something

Ask Belle 最有教学价值的一部分，反而是这些真实 bug。它们把“功能做出来”和“产品能用”之间的差距暴露得很清楚。

Hidden panel still occupied layout

面板虽然带了 hidden，但 CSS 把它重新拉回了布局里，导致 launcher 跑位。修法是显式收紧 [hidden] 和 panel 定位方式。

Chat panel slipped under the masthead

面板从 launcher 往上长，视口一矮就会钻到导航条下面。后来改成固定定位，并给它留出导航下方的安全顶部。

Answer scroll anchored to the wrong place

用 offsetTop 算滚动时，锚点相对于整个面板而不是消息容器，结果回答后用户看不到自己刚问的问题。

Contact chips were stretched by the wrong grid row

这不是“按钮太大”那么简单，而是 contacts 被放进了会拉伸的 1fr 行里，导致整排被撑大并挤压消息区。

Module 4

Hands-On Labs

如果把这个页面当成 workshop，下面就是最合理的 lab sequence。每个 lab 都围绕 Ask Belle 的真实实现切一刀。

Lab 1

Write your own scope contract: what your assistant can answer, what it should refuse, and what links it may safely surface.

Lab 2

Create a tiny knowledge JSON with 5 entries and wire a fallback mode that can answer without touching the model.

Lab 3

Add the widget shell to your own static site and make open, close, send, reset, and click-outside work first.

Lab 4

Deploy a serverless /api/chat endpoint with grounding, CORS, payload limits, and environment variables.

Lab 5

Fix one layout bug, one scroll bug, and one responsive bug in your own widget before calling it “done.”

Lab 6

Run a launch checklist that covers billing, cache invalidation, origin checks, and both live and fallback paths.

Module 4.5

Launch Checklist

这一节是为了避免“看懂了，但还是不知道怎么收尾上线”。如果你在做自己的版本，按这个 checklist 走，基本不会漏关键点。

Before backend

  • 主站里已经有清晰的知识来源页面，而不是空白 landing page。
  • 你已经准备了一个轻量知识 JSON。
  • widget 在没有模型的情况下也能打开、关闭、发送和 reset。
  • suggested prompts 已经反映出真实边界。

Before production

  • 后端已设置 OPENAI_API_KEY、ALLOWED_ORIGIN、knowledge base 地址。
  • 确认 OpenAI billing / credits 可用。
  • 确认 CORS 拒绝了非本站 origin。
  • 确认超长消息和高频请求会被后端收住。

UX smoke test

  • 点击 launcher 能打开，点击页面别处会收起。
  • Esc 可关闭，焦点不会乱跳。
  • 回答生成后，视图会回到用户问题附近。
  • 面板不会被导航栏压住，也不会把页面其他按钮挡死。

What to document

  • assistant 的回答范围和 fallback 策略。
  • 后端部署位置和最小环境变量。
  • 知识库文件如何更新。
  • 你为这个项目修过的三个真实 bug。
Capstone

Build Your Own Grounded Site Assistant

最终作业不是“做一个聊天框”，而是做一个可以上线的 bounded assistant。它应该有清晰范围、真实知识源、稳定 fallback、 以及能经受真实视口和真实部署环境的前端行为。

Big takeaway

在这个代码库里，真正有教学价值的不是单独一段 API 调用，而是完整链路：静态站约束、组件注入、行为设计、后端边界、上线后的 bug 处理，以及如何把一个“能跑”的 AI feature 收成一个“可用”的产品组件。

Deliverable checklist

  • bounded question scope
  • frontend widget integrated into an existing site
  • separate serverless backend
  • grounded retrieval or curated context
  • fallback mode
  • at least 3 real UI bug fixes
